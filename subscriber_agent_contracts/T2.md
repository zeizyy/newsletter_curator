# T2 Contract: Audience-Aware Newsletter Persistence For Personalized Delivery

## Goal
Persist and reuse daily newsletters by `(newsletter_date, audience_key)` so personalized delivery groups can safely cache their own digest variants without colliding with each other or with the legacy generic digest.

## In Scope
- Make `daily_newsletters` audience-aware while preserving existing telemetry and tracked-link relationships.
- Keep the legacy non-personalized path on the default audience key.
- Persist personalized delivery groups under their `profile_key`.
- Reuse cached personalized newsletters on later runs when the same audience key is requested again.
- Expose enough result metadata for the evaluator to verify which persisted audience variant was reused.

## Out Of Scope
- Preview personalization.
- Buttondown metadata sync.
- Subscriber self-serve preference editing.
- Reworking preview-generation locking beyond the existing default audience behavior.

## Required Behavior
1. `repository.get_daily_newsletter()` and `repository.upsert_daily_newsletter()` must support an `audience_key`, with `"default"` as the backward-compatible default.
2. The persistence model must allow multiple newsletter rows for the same `newsletter_date` when `audience_key` differs.
3. Existing default preview and non-personalized delivery must keep reading and writing the `"default"` audience key without changing caller behavior.
4. Personalized delivery groups must persist under their `profile_key` and may only reuse a cached newsletter whose `audience_key` exactly matches that group.
5. Two recipients sharing one `profile_key` must still reuse one persisted newsletter row.
6. Two different `profile_key` values on the same day must produce separate persisted rows and separate cached reuse behavior.
7. Telemetry and tracked-link persistence must keep working because they still point at `daily_newsletter_id`.
8. The top-level grouped delivery result must expose, for each `delivery_group`, at least:
   - `profile_key`
   - `audience_key`
   - `daily_newsletter_id`
   - `cached_newsletter`
   - `recipients`
9. Personalized delivery must stop bypassing persistence once this task is complete.

## Implementation Notes
- Use `"default"` as the compatibility audience key for the legacy generic digest.
- For personalized groups, use the existing `profile_key` as `audience_key`.
- `daily_newsletters` cannot be upgraded with only `ALTER TABLE ADD COLUMN` because the current table-level `UNIQUE(newsletter_date)` constraint would still block multiple rows per day. Rebuild the table with the new composite uniqueness instead.
- Preserve existing `daily_newsletters.id` values during migration so `newsletter_telemetry`, `tracked_links`, `newsletter_open_events`, and `newsletter_click_events` remain valid.

## Tests Required
Add `tests/integration/test_personalized_newsletter_cache_keys_by_profile.py` covering:
- a first personalized run persists one row per unique audience/profile key for the same day
- two recipients sharing a profile reuse the same persisted row
- a second personalized run with source data removed still succeeds from the cached audience-specific rows
- different profile keys do not reuse each other’s cached body/html
- the legacy default path still reads the default audience row

## Test Command
`uv run pytest tests/integration/test_personalized_newsletter_cache_keys_by_profile.py -q`

## Evaluator Checks
- The integration test proves same-profile reuse and cross-profile isolation with exact persisted row counts and audience keys.
- The implementation preserves the default non-personalized cache path.
- The migration preserves existing `daily_newsletter_id` foreign-key relationships instead of orphaning telemetry or tracked links.
- The grouped delivery result includes audience-aware cache metadata for each group.
