# T2 Contract: Audience-Aware Newsletter Persistence For Personalized Delivery

## Goal
Persist and reuse daily newsletters by `(newsletter_date, audience_key)` so personalized delivery groups can cache and reuse the correct digest variant without colliding with each other or with the legacy generic digest.

## In Scope
- Extend daily newsletter persistence to support multiple newsletter variants per date.
- Keep the legacy non-personalized path on a stable `audience_key = "default"`.
- Persist personalized delivery groups under their `profile_key`.
- Reuse cached personalized variants on later runs when the same profile reappears.
- Preserve telemetry and tracked-link relationships through the audience-aware persistence change.
- Expose enough per-group metadata to prove correct cache reuse in tests.

## Out Of Scope
- Preview personalization.
- Admin UI changes for multiple audience variants.
- Buttondown metadata sync.
- Changes to subscriber grouping semantics from `T1`.

## Required Behavior
1. `daily_newsletters` persistence must support multiple rows for the same `newsletter_date` as long as `audience_key` differs.
2. The non-personalized path must continue to read and write the generic digest under `audience_key = "default"`, so existing preview and history behavior still targets the default audience.
3. Personalized delivery groups must pass their `profile_key` as `audience_key` when reading or writing cached newsletters.
4. Two recipients with the same effective profile must reuse the same persisted newsletter row on subsequent runs.
5. Two different profile groups on the same date must not share one persisted newsletter row.
6. Personalized cache rows must not collide with a pre-existing generic `default` digest from the same date.
7. Existing databases that already contain `daily_newsletters` rows without `audience_key` support must migrate in place so those rows become `audience_key = "default"` while preserving their `id` values and dependent telemetry / tracked-link records.
8. The delivery result for each personalized group must expose at least:
   - `audience_key`
   - `daily_newsletter_id`
   - `cached_newsletter`
   - `profile_key`
9. Existing repository callers that do not request a specific audience must keep default-audience behavior.
10. `list_daily_newsletters(...)` must continue to return only the default audience unless a caller explicitly opts into all audiences.

## Implementation Notes
- Prefer evolving `daily_newsletters` rather than adding a second newsletter-content table, because telemetry and tracked links already key off `daily_newsletter_id`.
- The migration may rebuild `daily_newsletters` in place if needed, but it must preserve existing row ids for dependent telemetry tables.
- Keep `get_daily_newsletter(...)` defaulting to the generic audience so preview and current admin views remain stable.
- If `list_daily_newsletters(...)` changes, keep existing callers on default-audience semantics unless they explicitly opt into all audiences.

## Tests Required

- a first personalized run stores one row per unique audience/profile on the same date
- a second run with source data removed reuses the correct cached audience variants
- two recipients sharing a profile reuse the same `daily_newsletter_id`
- two distinct profiles get different `daily_newsletter_id` values
- a generic default digest does not get reused for a personalized audience key

Add `tests/integration/test_daily_newsletter_audience_key_migration_preserves_telemetry.py` covering:
- migrating an old `daily_newsletters` table preserves the old row id
- the migrated row is addressable as `audience_key = "default"`
- existing newsletter telemetry / tracked-link records still reference the migrated row successfully
