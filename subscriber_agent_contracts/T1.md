# T1 Contract: Subscriber-Profile Resolution And Grouped Personalized Delivery

## Goal
Ship subscriber-level `persona` and `preferred_sources` configuration that affects delivered digests end to end, while preserving the existing single-digest cached path when no personalized subscriber overrides are active.

## In Scope
- Add config support for top-level `subscribers` entries with:
  - `email`
  - optional `persona.text`
  - optional `preferred_sources`
- Normalize resolved recipients into subscriber profile objects.
- Group recipients by effective profile during delivery.
- Apply per-subscriber preferred-source filtering on top of globally enabled sources.
- Keep the current cached-newsletter path only for the non-personalized default flow.

## Out Of Scope
- Audience-aware newsletter persistence or cache-key changes.
- Preview personalization.
- Admin UI for editing subscriber preferences.
- Buttondown metadata sync for preferences.

## Required Behavior
1. A config with only legacy recipients must still deliver through the existing single-digest path.
2. Recipient membership continues to come from the existing sources in this order:
   - explicit dry-run recipient override
   - Buttondown recipients when configured
   - `email.digest_recipients`
   Top-level `subscribers` entries do not add recipients on their own; they only override profile settings for matching resolved emails.
3. A subscriber override must be matched by normalized email and produce an effective profile with:
   - `email`
   - `persona_text`
   - `preferred_sources`
   - `profile_key`
4. `preferred_sources` matching is exact after trim/lowercase against candidate `source_name`. A missing or empty list means no subscriber-specific narrowing.
5. Global repository source enable/disable still runs first. `preferred_sources` can only narrow the already-allowed candidate set; it cannot re-enable globally disabled sources.
6. Subscribers sharing the same effective profile must receive the same generated digest without duplicate generation work.
7. Personalized delivery must expose `delivery_groups` in the top-level result with one entry per unique `profile_key`, including at least:
   - `profile_key`
   - `recipients`
   - `sent_recipients`
   - `preferred_sources`
   - `persona_text`
   This metadata is the observable signal the evaluator will use to verify grouping.
8. Subscribers with different personas or preferred sources must be able to receive different digest content when the candidate pool differs.
9. Personalized delivery must not incorrectly reuse the existing single-day cached newsletter.

## Implementation Notes
- Keep recipient resolution logic centralized rather than scattering subscriber parsing through delivery code.
- Use the existing global persona as the fallback default when a subscriber has no explicit persona.
- When personalized delivery is active, it is acceptable for this sprint to bypass persisted daily newsletter reuse.

## Tests Required
Add `tests/integration/test_delivery_personalizes_by_subscriber_profile.py` covering:
- legacy delivery still works unchanged when no subscriber overrides exist
- two subscribers with different profiles receive different digest content
- two subscribers sharing a profile receive identical digest output
- `result["delivery_groups"]` has one entry per unique profile, so a shared-profile pair proves grouped generation happened once
- preferred_sources narrows the delivered source pool using exact trim/lowercase `source_name` matching

## Test Command
`uv run pytest tests/integration/test_delivery_personalizes_by_subscriber_profile.py -q`

## Evaluator Checks
- The feature is backward compatible for non-personalized delivery.
- The integration test would fail if persona or preferred_sources were ignored.
- The implementation does not accidentally serve one cached digest to all personalized recipients.
- The repo remains ready for a later audience-aware persistence task.
