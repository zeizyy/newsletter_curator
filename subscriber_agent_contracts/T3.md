# T3 Contract: Document Subscriber Personalization And Operator Caveats

## Goal
Document the shipped subscriber-personalization behavior so operators can configure per-recipient persona and preferred-source overrides without reading the implementation.

## In Scope
- Document the top-level `subscribers` config shape in `README.md`.
- Add an example `subscribers` block to `config.yaml`.
- Explain how recipient resolution interacts with subscriber overrides.
- Call out the current preview/history/persistence caveats for personalized audiences.
- Document the exact matching semantics for `preferred_sources`.

## Out Of Scope
- Preview personalization.
- Admin UI changes for editing subscriber overrides.
- Buttondown metadata sync for subscriber preferences.
- Any runtime delivery or persistence behavior changes.

## Required Behavior
1. The docs must state that delivery recipients are still resolved in this order:
   - explicit dry-run recipient override
   - Buttondown subscribers when `BUTTONDOWN_API_KEY` is configured
   - `email.digest_recipients`
2. The docs must state that top-level `subscribers` entries do not add recipients on their own; they only override profile settings for matching resolved email addresses.
3. The docs must show a valid `subscribers` example with:
   - `email`
   - optional `persona.text`
   - optional `preferred_sources`
4. The docs must explain that `preferred_sources` matching is exact after trim/lowercase against candidate `source_name`.
5. The docs must explain that subscriber `preferred_sources` only narrow the already-enabled global source set; they do not bypass global source filtering.
6. The docs must explain that non-personalized delivery and preview continue using the default audience path.
7. The docs must call out that preview, newsletter history, and analytics currently remain focused on the default audience rather than enumerating personalized variants.

## Implementation Notes
- Keep the README guidance operator-facing instead of describing internal helper names.
- Use the sample `config.yaml` to provide a copyable starting point for `subscribers`.
- Keep the caveat wording aligned with the current shipped behavior from `T1` and `T2`.
